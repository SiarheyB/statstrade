# План: Orderbook Heatmap (как ClusterBtc) — Вариант B

> Полноценная тепловая карта лимитных ордеров с историей, как на clusterbtc.com / Bookmap:
> X — время, Y — цена, яркость ячейки — объём лимитных заявок на уровне; «стены» гаснут,
> когда цену через них проливают. Плюс слайдеры фильтрации мелких заявок.

## Решения (зафиксированы)
- **Вариант B** — полноценный heatmap с историей (не синтетика, в отличие от текущего `liqmap`).
- Отдельный **collector-сервис вне Vercel** (Railway/Fly/VPS) — постоянный WebSocket.
- Хранилище — та же **Neon Postgres**, новые таблицы.
- Фронт — новый раздел в текущем Next.js, рендер на canvas (переиспользуем подход `liqmap`).
- **Старт:** Binance Futures, пара **BTCUSDT** (дефолт, можно сменить).
- **Шаг ценового бина:** **$25** для BTC (дефолт, можно сменить).
- Collector живёт в подпапке репозитория `collector/` (отдельный package.json, отдельный деплой).

## Архитектура
```
Биржи (WS depth) → Collector (локальный стакан + ресинк) → Neon Postgres (агрегаты) → Next.js /api/orderflow → Canvas heatmap (+ SSE live)
```

## Риски / на что закладываться
- **Целостность стакана** — главный источник багов: ресинк при разрыве sequence обязателен.
- **Объём БД** — без retention/rollup Neon распухнет; параметры считать заранее.
- **Стоимость** хостинга collector (постоянный процесс).
- Рейт-лимиты REST при частых ресинках; выравнивание времени между биржами при агрегации.
- ToS/лицензии бирж на публичные WS при расширении.

### 2026-06-27 — Этап 1 выполнен

  `prisma/migrations/20260627120000_orderbook_snapshot/migration.sql`

- Параметры (ENV, дефолты): BIN_SIZE=$25, SNAPSHOT_MS=2000, DEPTH_PCT=0.02,

  корректно видны крупные стены (напр. bid 59200 ≈ 300 BTC). БД-данные оставлены для разработки Этапа 2.
- ⚠️ Объём: при текущих параметрах ~10-15M строк/неделю (~1 ГБ). Для Neon free тесновато —
